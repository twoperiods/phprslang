# PHPRS 生产环境就绪度评估报告

> 评估日期: 2026-05-07  
> 评估版本: master@84de186 (线程池优化后)  
> 评估范围: PHPRS 编译器 + MVC 框架 + C 运行时

---

## 总体评估: 尚未达到生产环境部署标准

**结论: PHPRS 当前适合原型开发、教学演示和内部工具，不建议部署到面向公众的生产环境。**

综合评分: **4/10**（10 = 生产就绪）

## 2. 详细评估

### 2.1 性能 (8/10) ✅

**优势:**
- 编译模式 27,000+ QPS（c=50），接近 Go 标准库水平
- 8 线程池并行处理，CPU 多核利用
- 166KB 轻量二进制，启动 < 1ms
- 内存占用仅 7.3MB

**不足:**
- 无 HTTP keep-alive（每请求一次 TCP 连接）
- 单 accept 线程在 c=200+ 时成为瓶颈
- 无连接池复用

### 2.2 内存安全 (3/10) ❌

**严重问题:**

| 问题 | 详情 |
|------|------|
| 手动内存管理 | 105 次 malloc/calloc，94 次 free — 存在 11 处潜在泄漏 |
| 无内存池 | 每次请求分配/释放字符串，高并发下碎片化严重 |
| sprintf 使用 | 4 处不安全 sprintf（虽多数已用 snprintf） |
| 字符串拼接 | 大量 strdup + realloc 拼接，长连接场景下内存无上限 |
| 无 Valgrind/ASan 验证 | 未进行内存检测工具的系统性测试 |

**风险场景:**
- 长时间运行（数天）后内存逐渐增长
- 恶意构造超长请求导致 OOM
- 并发竞态下 double-free

### 2.3 安全性 (4/10) ❌

**已有:**
- CSRF token 验证
- CORS 配置（非通配符）
- Session ID 使用 `random_bytes`
- 基本输入校验（session_valid_id）
- htmlspecialchars XSS 防护

**缺失:**

| 缺失项 | 风险等级 | 说明 |
|--------|---------|------|
| HTTPS/TLS 服务端 | 严重 | 运行时仅支持 TLS 客户端连接，服务端无 TLS |
| 请求体大小限制 | 严重 | 无 max body size，可被大文件攻击耗尽内存 |
| 请求超时 | 高 | 慢速攻击 (Slowloris) 可耗尽线程池 |
| HTTP Header 注入防护 | 高 | 响应头拼接未过滤 \r\n |
| SQL 注入防护 | 高 | 无参数化查询（虽当前无 SQL 数据库） |
| 安全响应头 | 中 | 缺少 X-Frame-Options, CSP, HSTS 等 |
| 密码哈希 | 中 | 无 bcrypt/argon2，仅有 md5/sha1 |
| IP 黑名单/WAF | 中 | 速率限制是内存级的，重启后丢失 |

### 2.4 可观测性 (2/10) ❌

**完全缺失:**
- 无 access log（请求日志）
- 无 error log（错误日志）
- 无结构化日志（JSON logging）
- 无 metrics 端点（Prometheus/StatsD）
- 无分布式追踪（OpenTelemetry）
- 无健康检查端点（/health, /ready）
- 无请求 ID 关联

**影响:** 生产环境中出现问题时无法排查，无法设置告警，无法观察系统状态。

### 2.5 错误处理 (3/10) ❌

| 问题 | 详情 |
|------|------|
| Rust 侧 98 处 unwrap/expect | 编译器阶段的 panic 不会影响运行时，但 build 命令可能崩溃 |
| C 运行时用 setjmp/longjmp | 非结构化异常，栈展开不可靠，可能泄漏资源 |
| 无 panic recovery | 单个请求的 C 层段错误会终止整个进程 |
| 信号处理缺失 | 无 SIGTERM/SIGINT 处理器，无法优雅关闭 |
| 无请求隔离 | 一个请求的错误（如栈溢出）影响所有并发请求 |

### 2.6 运维能力 (2/10) ❌

**缺失:**

| 能力 | 状态 |
|------|------|
| 优雅关闭 (graceful shutdown) | ❌ kill -9 是唯一关闭方式 |
| 配置文件 | ❌ 端口/线程数硬编码在源码中 |
| 环境变量支持 | ❌ 无 env var 读取 |
| 热重载 / 零停机部署 | ❌ 必须重启进程 |
| 进程管理 | ❌ 无 daemon 模式，无 PID 文件 |
| 日志轮转 | ❌ 无日志输出 |
| 资源限制 | ❌ 无最大连接数/最大请求体限制 |
| 系统化部署 | ❌ 无 systemd/Docker/k8s 配置 |

### 2.7 测试覆盖 (3/10) ❌

| 测试类型 | 状态 | 详情 |
|----------|------|------|
| 单元测试 | ⚠️ 有限 | 20 个测试，覆盖词法/语法/类型检查基础路径 |
| 集成测试 | ❌ 无 | 无端到端 HTTP 请求测试 |
| 压力测试 | ✅ 有 | ab 压测通过，但非自动化 |
| 模糊测试 | ❌ 无 | 解析器/HTTP 解析器未经 fuzz |
| 内存测试 | ❌ 无 | 无 Valgrind/ASan/MSan 验证 |
| 安全测试 | ❌ 无 | 无渗透测试/OWASP 扫描 |
| 并发测试 | ❌ 无 | 无 ThreadSanitizer 验证 |
| 回归测试 | ❌ 无 | 无 CI/CD 管道 |

### 2.8 数据持久化 (2/10) ❌

- 数据库为 JSON 文件读写，无事务、无并发安全
- Session 为文件存储，多实例无法共享
- Redis 控制器为桩代码（无真实 Redis 连接）
- 无连接池、无 ORM、无迁移工具

## 3. 风险矩阵

| 风险 | 可能性 | 影响 | 优先级 |
|------|--------|------|--------|
| 内存泄漏导致 OOM | 高 | 严重（服务不可用） | P0 |
| 无 TLS → 数据泄露 | 确定 | 严重（合规问题） | P0 |
| 慢速攻击耗尽线程池 | 高 | 严重（DoS） | P0 |
| 超大请求体 OOM | 高 | 严重（DoS） | P0 |
| 段错误导致进程退出 | 中 | 严重（服务中断） | P1 |
| 无日志无法排障 | 确定 | 高（运维困难） | P1 |
| 文件 DB 并发损坏 | 中 | 高（数据丢失） | P1 |
| Header 注入 | 低 | 中（安全漏洞） | P2 |

## 4. 适用场景判断

### ✅ 适合的场景

| 场景 | 理由 |
|------|------|
| 学习/教学 | PHP-like 语法 → C 编译的有趣演示 |
| 原型开发 | 快速搭建 API 原型验证想法 |
| 内部工具（无公网暴露） | 性能好、部署简单、单文件 166KB |
| CTF/竞赛项目 | 展示编译器技术 |
| 嵌入式/IoT 控制面板 | 极低资源占用 |
| CLI 工具的 Web 界面 | 简单场景无需完整框架 |

### ❌ 不适合的场景

| 场景 | 缺失的关键能力 |
|------|---------------|
| 面向公网的 Web 服务 | TLS、WAF、安全头、请求限制 |
| 用户数据处理 | 数据库、事务、加密存储 |
| 电商/支付 | PCI-DSS 合规、审计日志 |
| 高可用服务 | 优雅关闭、健康检查、自动恢复 |
| 微服务架构 | 服务发现、链路追踪、配置中心 |
| 团队协作项目 | 包管理、依赖管理、CI/CD |

## 6. 短期替代方案

如果需要立即将 PHPRS 应用部署到生产环境，建议采用以下架构缓解风险：

```
Internet
    │
    ▼
┌─────────────┐
│   Nginx     │ ← TLS 终止、请求限制、access log、安全头
│  (反向代理) │ ← 请求体限制、超时控制、WAF 规则
└──────┬──────┘
       │ HTTP (内网)
       ▼
┌─────────────┐
│ PHPRS 二进制 │ ← 仅处理业务逻辑
│ (127.0.0.1) │ ← 不直接暴露公网
└─────────────┘
```

**此方案可解决:**
- TLS/HTTPS（Nginx 处理）
- 请求大小限制（Nginx `client_max_body_size`）
- 超时（Nginx `proxy_read_timeout`）
- Access log（Nginx 自带）
- 安全头（Nginx 配置添加）
- 速率限制（Nginx `limit_req_zone`）
- 进程管理（systemd 守护 PHPRS 进程）

**仍然无法解决:**
- 内存泄漏（长时间运行后 OOM）
- 段错误导致进程退出（需 systemd 自动重启）
- 无结构化日志
- 文件 DB 并发安全

---

## 7. 结论

PHPRS 作为编译器项目技术实现出色 — 从 PHP-like 语法到 166KB 原生二进制的编译链路完整，性能表现优异。但作为 Web 框架的生产部署，缺少太多基础设施层面的能力。

**核心差距:** 不是性能问题（性能已达标），而是**可靠性、安全性和可运维性**的系统性缺失。

**建议:**
1. **短期内**：如需部署，必须放在 Nginx 反向代理后，且仅用于内部/低风险场景
2. **中期**：按 Phase 1-2 路线图补齐安全底线和稳定性
3. **定位调整**：考虑将 PHPRS 定位为「嵌入式/边缘计算 Web 引擎」而非通用 Web 框架，发挥其 166KB 极轻量的优势
